## 1. Story Summary

QINDU-0008 delivers encrypted persistent storage for token↔PII mappings (`internal/crypto`, `internal/vault`, `internal/session`). The vault uses AES-256-GCM encryption via the Go stdlib, bbolt for on-disk storage, and per-user isolation via Windows SID lookup. TTL-based auto-expiration respects data minimization. The vault is designed as an optional `TokenPersister` subscriber to the Tokenizer — operating identically with or without it (nil = memory-only).

**Key design decisions**: per-conversation TTL (7-day default), async writes (proxy never blocks on disk), bbolt keys are plaintext (provider/UUID/token-ID only), values are ciphertext, metadata is plaintext (enables UI browsing and TTL sweeps without decryption).

## 2. Data Processed — PII Categories

The vault stores the following categories when wired to the Tokenizer (not yet wired — deferred to QINDU-0009):

| Category | Detection | Stored in vault? | Encryption |
|----------|-----------|-----------------|------------|
| Email addresses | PII engine | Yes — value encrypted | AES-256-GCM |
| Phone numbers | PII engine | Yes — value encrypted | AES-256-GCM |
| Credit card numbers | PII engine | Yes — value encrypted | AES-256-GCM |
| IBAN | PII engine | Yes — value encrypted | AES-256-GCM |
| Names (person) | PII engine | Yes — value encrypted | AES-256-GCM |
| SSN / fiscal IDs | PII engine | Yes — value encrypted | AES-256-GCM |
| Physical addresses | PII engine | Yes — value encrypted | AES-256-GCM |
| Passport numbers | PII engine | Yes — value encrypted | AES-256-GCM |
| IP addresses | PII engine | Yes — value encrypted | AES-256-GCM |

**Metadata stored in plaintext** (non-PII): provider name, conversation UUID (proxy-generated, not linked to any external identity), timestamps, PII type counts (e.g., `pii_types: ["EMAIL", "IBAN"]`), and status flags. This design is approved per DD-4 — the plaintext metadata enables prefix scans for the future UI and TTL enforcement without requiring decryption of all values.

**Tokens** (e.g., `<<EMAIL_1>>`) are used as bbolt keys and are NOT PII — they are surrogate identifiers generated by the Tokenizer.

## 3. Purpose — Why This Processing Is Necessary

The vault serves three purposes defined in the story:

1. **Cross-session conversation continuity**: Users restarting their browser or machine can resume conversations where PII tokens are rehydrated from the vault.
2. **Conversation browsing (QINDU-0016)**: A future UI will display past conversations and their metadata without exposing raw PII.
3. **Data lifecycle management**: TTL-based auto-expiration ensures PII is automatically purged, implementing data minimization by default.

Without the vault, PII token↔value mappings exist only in memory and are lost on proxy restart. The user would need to re-enter PII in every new session — degrading the proxy's core value proposition of transparent PII protection.

---

## 4. Minimization Basis — Why Less Would Not Work

**Why persist rather than memory-only?** Memory-only storage loses all mappings on proxy restart. Without persistence, the user would need to re-enter PII values for every new browser session — the proxy cannot rehydrate tokens from AI responses without knowing what the token represents.

**Why per-conversation rather than flat?** Flat storage would require decrypting all entries to enforce TTL or browse conversations. The conversation-scoped schema with plaintext metadata enables TTL sweeps, UI browsing, and per-conversation purging without bulk decryption.

**Why 7-day TTL?** 7 days (168h) is the default, configurable down to 24h. This balances usability (most conversations span a working day or week) with data minimization (infinite retention is possible via `ttl: 0` but logs a WARNING at startup).

**Why metadata in plaintext?** Storing `pii_count`, `pii_types`, and timestamps in plaintext is the minimal set of metadata needed for:
- TTL enforcement (needs `created_at`)
- UI conversation listing (needs provider, timestamps, PII type summary)
- This metadata contains no PII values — only type names and counts.

**Why AES-256-GCM key file rather than DPAPI?** The vault must be portable across Windows, Linux, and macOS. DPAPI is Windows-only. AES-256-GCM via Go stdlib works identically on all platforms with hardware acceleration (AES-NI, ARM Crypto). The CA's DPAPI encryption remains untouched (`ca_windows.go`).

## 5. Rights and Freedoms Risks

### Risk 1: Vault data survives uninstall without explicit opt-in
- **Finding**: The QEMU test report confirms `C:\ProgramData\Qindu\` (CA files) persists after uninstall unless `DELETEDATA=1` is passed. However, vault files (user PII data) are **not** created in the current sprint because the vault is not wired to the proxy. When wired (QINDU-0009), per-user vault.db files in `%LOCALAPPDATA%\Qindu\` may also need an uninstall cleanup path.
- **Mitigation**: Per-user data cleanup during uninstall is complex (the MSI runs as SYSTEM, cannot enumerate all user profiles). This must be addressed in a future installer sprint with a cleanup script or documentation.
- **Severity**: Low (vault not yet wired).

### Risk 2: `vault.key` persistence after data erasure
- **Finding**: The peer review (previous round, PR-106) noted that `vault.key` is not shredded on disk after a PurgeAll or TTL expiry. An attacker with the key file could decrypt old bbolt snapshots.
- **Mitigation**: Key file is `0600` permissioned and resides in a per-user directory protected by OS ACLs. For V1, this is acceptable. A future "right to erasure" operation should include key rotation + secure deletion.
- **Severity**: Low (requires disk access + key file access).

### Risk 3: `slog.Default()` fallback in vault code
- **Finding**: `vault.New()` (line 77) and `NewVaultManager()` (line 50) silently fall back to `slog.Default()` when a nil logger is passed. This bypasses the caller's structured logging chain.
- **Mitigation**: Every log call in the vault code includes `"pii_values_logged", false`. No PII values are ever logged. File paths are redacted via `redactHomePath()`. The operational metadata logged (vault open, close, sweep counts) is PII-free.
- **Status**: Acknowledged design choice per peer review PR-103. Non-blocking — no PII exposure risk.
- **Severity**: None (PII-free operational logs only).

### Risk 4: CA private key `/tmp` fallback on Unix
- **Finding**: `getCADir()` falls back to `/tmp/qindu-ca` when both `PROGRAMDATA` and `UserHomeDir` are unavailable. On multi-user Unix systems, `/tmp` is world-readable — exposing the CA private key to other users would compromise all intercepted HTTPS traffic.
- **Mitigation**: V1 targets Windows only. On Windows, `%PROGRAMDATA%` is always set. This fallback is unreachable in production. Documented for future Linux porting.
- **Severity**: Low (theoretical, not reachable on Windows).

### Risk 5: No access control on vault database beyond filesystem permissions
- **Finding**: The vault.db file is protected by filesystem ACLs (user profile directory on Windows, 0700 directory on Unix). There is no application-level password or key derivation from a user secret — the `vault.key` is a random 32-byte file.
- **Mitigation**: This is by design (DD-9, DD-12). The threat model assumes single-user machines. Multi-user isolation on Windows service mode is handled via SID-based path resolution.
- **Severity**: Low.

## 11. Current Sprint Scope Analysis

The most significant finding is what the vault is **NOT** doing in this sprint:

- **Vault not wired to proxy**: The vault infrastructure (`internal/crypto`, `internal/vault`, `internal/session`) exists as thoroughly-tested library code but is **not connected** to the proxy binary. The proxy operates in memory-only mode (`TokenPersister = nil`). This is **the most privacy-preserving stance** — no PII is being persisted to disk at all. This is correct for the sprint scope (vault wiring is QINDU-0009).

- **`initVault()` fully removed**: The previous blocking issue (QEMU F3 — phantom vault in `C:\Windows\ServiceProfiles\LocalService\AppData\Local\Qindu\`) is resolved. The proxy no longer creates any vault at startup.

- **SID lookup infrastructure ready but unused**: The Windows SID lookup code with `TOKEN_IMPERSONATE` flag exists in `internal/session/lookup_windows.go` but is not called from the proxy binary. It will be activated in QINDU-0009 for per-user vault resolution at connection time.

## 15. Verdict

### ✅ **PASS**

**Rationale**:

1. **No PII exposure in this sprint**: The vault is library-only, not wired to the proxy. No PII is being persisted to disk. The proxy operates in memory-only mode — identical to pre-vault behavior.

2. **Privacy-positive changes**: The removal of `initVault()` fixes the phantom vault in the service profile. The `TOKEN_IMPERSONATE` flag enables correct per-user isolation. The `resolveCRLDP()` function removes hardcoded Windows paths.

3. **All code respects ADR-008**: Every log call in vault/crypto/session/tls code includes `pii_values_logged: false`. File paths are redacted. No PII values in any error messages.

4. **Test fixtures are clean**: All test data uses reserved domains (`example.com`), fictional phone numbers (`+1-555-0100`), and synthetic credit card numbers. Zero real PII.

5. **ADR-003/004/008 compliance**: CA DPAPI untouched. Interceptor interface unchanged. Structured logging respected.

6. **All three previous QEMU blockers resolved**: No phantom vault, no leaked vault files, ProgramData behavior is intentional.

7. **No telemetry, analytics, tracking, user accounts, or external network calls**: Verified in all changed files.

**The vault infrastructure is ready for wiring in QINDU-0009. No blocking privacy or data protection issues found in the current sprint scope.**
