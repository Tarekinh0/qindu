# QINDU-HOTFIX-001: MSI Installer + TLS Leaf Cert Bugs

**Status**: DONE  
**Type**: Hotfix  
**Affected Sprints**: QINDU-0001 (TLS), QINDU-0002 (Installer)  
**Source**: QEMU VM integration test — `qemu-test-report.md`

## Bugs to Fix

All 4 bugs are CRITICAL — found by qemu-tester on real Windows VM.

### BUG-001: MSI compiled as 32-bit — wrong install path

**File**: `installer/wix/qindu.wxs`  
**Problem**: Missing `Platform="x64"` on `<Package>`. Windows Installer's WIN64DUALFOLDERS redirects to `C:\Program Files (x86)\` for 32-bit packages.  
**Fix**: Add `Platform="x64"` attribute to the `<Package>` element.

### BUG-002: Custom actions not wired into InstallExecuteSequence

**Files**: `installer/wix/qindu.wxs` (main), `includes/ca-trust.wxs`, `includes/firewall.wxs`  
**Problem**: Custom actions (ca-init, certutil -addstore Root, netsh advfirewall) are defined as `<CustomAction>` elements but not scheduled in `<InstallExecuteSequence>`. The MSI deploys files + registry + service but never generates the CA, installs it in the trust store, or creates firewall rules.  
**Fix**: Add `<Custom>` elements to `<InstallExecuteSequence>` in `qindu.wxs` for each custom action, with correct `Before`/`After`/`Not` conditions:
  - `CAInitNormal` / `CAInitUnsafe` → after `InstallFiles`, before service install
  - `CACheckTrustStore` → after CA init
  - `CAInstallTrustStore` → after `CACheckTrustStore`
  - `CAInstallChromePolicies` / `CAInstallEdgePolicies` → after `WriteRegistryValues` (or equivalent)
  - `CABlockExternal` / `CAAllowLoopback` → after `InstallFiles`
  - Uninstall cleanup actions → after `InstallInitialize` during removal

### BUG-003: Service start sequenced before CA generation

**Files**: `installer/wix/includes/service.wxs`, `installer/wix/qindu.wxs`  
**Problem**: Service start (`StartServices` or explicit `<ServiceControl Start="install">`) runs before or independently of CA generation custom actions. If CA doesn't exist, service fails → Error 1920 → rollback → Error 1603.  
**Fix**: Ensure service install/start happens AFTER CA generation in the sequence. The correct order is: InstallFiles → CAInit → CACheckTrustStore → CAInstallTrustStore → Firewall → InstallServices → StartServices.

### BUG-004: TLS leaf certs lack revocation extensions — schannel fails

**File**: `internal/tls/cert.go`  
**Problem**: Leaf certificates generated by Qindu's MITM CA lack CDP (CRL Distribution Point) and OCSP (AIA/Authority Information Access) extensions. Windows schannel attempts revocation check on the leaf cert during TLS handshake, finds no revocation information, and fails with `CRYPT_E_NO_REVOCATION_CHECK (0x80092012)`.  

**Root cause**: The leaf cert template in `CreateLeafCert` or equivalent function does not include:
- `CRLDistributionPoints` extension
- `AuthorityInfoAccess` with OCSP responder URL  
- Or alternatively, the cert should include `id-ce-basicConstraints` with `cA:FALSE` and no revocation extensions, plus a known CRL DP pointing to a local/inaccessible URL that schannel treats as "revocation not required"

**Fix options** (choose the appropriate one):
1. Add a dummy OCSP responder URL and CRL DP to the leaf cert template that points to localhost or a non-existent URL. Windows schannel will attempt the check, fail gracefully, and proceed (since the CA is in the trust store).
2. Set appropriate EKU (Extended Key Usage) and mark the cert as not requiring revocation check through proper extensions.
3. The simplest approach for a MITM proxy: include an OCSP AIA pointing to `http://localhost/ocsp` and a CRL DP pointing to `http://localhost/crl`. Since the CA is in the machine trust store, schannel should tolerate the unreachable revocation endpoints.

Test with: `curl --ssl-no-revoke -x http://127.0.0.1:8787 https://chatgpt.com/` should work as baseline (this disables revocation check client-side). Then remove `--ssl-no-revoke` — it should still work after the fix.

## Scope

- Fix `installer/wix/qindu.wxs`: add `Platform="x64"`
- Fix `installer/wix/qindu.wxs`: wire all `<Custom>` actions into `<InstallExecuteSequence>`
- Fix `installer/wix/includes/service.wxs` (if needed): ensure correct sequencing
- Fix `internal/tls/cert.go`: add revocation extensions to leaf certs
- Add test for cert revocation extensions (unit test checking leaf cert has expected extensions)
- Update CI if needed (WiX candle/light commands may need `-arch x64` flag)

## Acceptance Criteria

1. `msiexec /i Qindu-Installer-x64.msi /qn /norestart` installs successfully on Windows 10+ VM
2. Files deployed to `C:\Program Files\Qindu\` (NOT `Program Files (x86)`)
3. After install: CA present in trust store, service running, firewall rules active, registry policies set
4. CA Name Constraints present (`*.chatgpt.com`, `*.claude.ai`)
5. `curl -x http://127.0.0.1:8787 https://chatgpt.com/` returns HTML without `--ssl-no-revoke` flag
6. `curl -x http://127.0.0.1:8787 https://example.com/` tunnels successfully
7. `/health` and `/proxy.pac` respond correctly
8. Uninstall removes everything; `DELETEDATA=1` also removes `%PROGRAMDATA%\Qindu\`
9. All 148 existing tests still pass (`go test -race ./...`)
10. WiX build succeeds (`candle` + `light` produce valid MSI)
11. `go vet ./...` clean, `golangci-lint` clean
